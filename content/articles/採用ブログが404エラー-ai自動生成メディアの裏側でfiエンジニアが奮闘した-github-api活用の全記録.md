## 採用ブログが404エラー…正直、顔面蒼白でした

皆さん、こんにちは。Futuristic Imagination LLC 代表の佐藤琢也です。

いきなりですが、実は先日、当社の採用ブログが突如「404 Not Found」になってしまうという、まさかの事態が発生しました。正直に言うと、背筋が凍りましたね。採用候補者の方に「この会社で働いてみたい」と思っていただくための大切な窓口が、エラー画面になってしまっている。これは由々しき事態です。

弊社の事業は、AIを活用したオウンドメディアの構築や、Gemini APIを使ったパイプライン開発受託。つまり、AIによるコンテンツ自動生成を核としています。私自身、現在1人で11ものサイトを自動運営しており、今回の採用ブログもその一つ。AIが生成した記事をVercel Cronで定期的に更新し、GitHubリポジトリにコミットするという、完全に自動化されたフローで運用しているんです。

だからこそ、手作業で何かをミスしたわけじゃない。システムは完璧なはず、という自負がありました。それなのに404。何が原因なのか、まさに頭の中を駆け巡りました。しかし、問題解決はエンジニアの本能。このピンチを乗り越えるべく、私はすぐに原因究明と解決策の立案に取り掛かりました。

## 自動生成メディアの闇とGitHub APIの壁

404エラーの原因は、実は自動生成プロセスとGitHubの連携の深部にありました。具体的には、GitHub APIを使ったコンテンツの更新と、Vercelによるデプロイのタイミング、そしてパス解決のロジックが複雑に絡み合っていたのです。

私たちのシステムでは、AI（Google Gemini）が生成した記事コンテンツを、Next.jsプロジェクトの特定のディレクトリに配置し、それをGitHubにコミットすることで、Vercelが自動的にビルド＆デプロイを行う仕組みになっています。記事のURLはファイルパスに基づいて動的に生成されるため、ファイルが正しく存在しないと即座に404エラーとなってしまう。

問題の核心は、GitHub APIを使ってファイルを更新・削除する際のアプローチにありました。当初の実装では、単に新しいコンテンツをアップロードするだけでなく、古いコンテンツを確実に削除し、GitHubのツリー構造とVercelのキャッシュが常に同期している状態を保つ必要がありました。しかし、GitHub APIは非常に強力である反面、その操作には細心の注意が必要です。特に、多数のファイルを一括で操作したり、ツリー構造を動的に変更したりする際には、APIのレートリミットや非同期処理の管理が重要になります。

私は元々SESのエンジニアとして様々なプロジェクトを経験してきましたが、AIスタートアップを立ち上げてからは、まさに「バイブコーディング」とでも言うべきスピード感で開発を進めてきました。厳密な設計よりも、まずは動かすことを優先し、その上で改善を重ねていくスタイルです。しかし、今回はそのアプローチが裏目に出た形でした。特にTypeScriptのStrict Mode環境下で、型安全性を保ちつつGitHub APIの複雑なレスポンスを扱うのは、想像以上に骨が折れる作業でしたね。

ビルドエラーが頻発し、Vercelのデプロイログとにらめっこする日々。既存記事の一括更新システムにも影響が出かねない状況で、いかに安定性と効率性を両立させるか。この課題は、私の「自動化を極めて重視する」という価値観と真正面から向き合うものとなりました。

## 奮闘、そして掴んだ「自動化」の光

この404エラー問題への解決策は、GitHub APIをより深く理解し、その特性を最大限に活かすことでした。私が辿り着いたのは、以下のステップを踏む、より堅牢なコンテンツ管理システムへの進化です。

1.  **GitHub Trees APIの徹底活用**: 単純なファイル操作ではなく、GitHubのGit Data APIにある「Trees API」を深く掘り下げました。これにより、コミット時にツリーオブジェクトを直接構築し、古いファイルを正確に削除しつつ、新しいファイルを追加する原子的な操作が可能になりました。これにより、Vercelがデプロイする際のファイルパスの不整合を根本的に解消することができたんです。
2.  **型安全性の強化（TypeScript Strict Mode）**: GitHub APIのレスポンスは非常に複雑ですが、TypeScriptのStrict Modeを徹底的に適用し、すべてのレスポンスを型で厳密に管理しました。これにより、APIの呼び出しミスや、レスポンスデータの不正な扱いによるバグを未然に防ぐことができました。デバッグの時間も大幅に短縮され、まさに「型が守ってくれる」感覚でしたね。
3.  **Vercel CronとDiscord連携の最適化**: AIによる記事生成とGitHubへのコミットはVercel Cronで自動実行されますが、このプロセスの成功・失敗をDiscordにリアルタイムで通知する機能をさらに強化しました。これにより、今回のような404エラーが発生した場合でも、即座に異常を検知し、対応を開始できる体制を確立しました。安定運用には、異常検知と迅速な対応が不可欠ですから。
4.  **リポジトリ構造の最適化とキャッシュ戦略**: 無駄なファイルを削減し、リポジトリ構造をシンプルにすることで、ビルド時間を短縮し、Vercelのキャッシュがより効果的に機能するように調整しました。これは、常にシステム全体の最適化と自動化を追求する私の哲学でもあります。

これらの取り組みを通じて、採用ブログは無事に復活しました。そして、この経験は私の「継続的な改善と成長」の燃料となりました。課題を乗り越えるたびに、システムはより強固に、より効率的になっていく。このサイクルこそが、Futuristic Imaginationの成長の源泉です。
