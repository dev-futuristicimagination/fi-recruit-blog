## 朝8時、牙を剥いた「API失効・デプロイ重複・トークン切れ」のトリプルパンチ

今回のトラブルは、複数の要因が奇妙なタイミングで重なったことで発生しました。

主力プロダクトである「補給不要の自販機型」採用コンテンツ自動化SaaS、そして自社メディアのSEO自動リライトパイプライン。これらはNext.js（Version 16系）と厳格なStrict Modeで制御されたTypeScript、そしてGemini 2.5 FlashなどのAPIを駆使して、サーバーレスで稼働しています。

その朝、起きていた事象は以下の3つでした。

1. **特定外部APIキーの突発的な失効**
2. **深夜のコンテンツ自動生成バッチと、リライト処理が重なったことによるVercelのデプロイキュー重複（ビルド詰まり）**
3. **連続したデータ処理に伴う、LLM（Gemini API）のトークン切れ**

どれか一つであれば、数分で対応可能な日常のノイズです。しかし、複数のプロジェクトでこれらが同時に発生したため、インフラの監視ダッシュボードは真っ赤に染まっていました。

一般的な受託会社や開発チームであれば、ここで「緊急会議」を開き、担当者をアサインし、原因をスプレッドシートに書き出し、役割分担を決めるところから始めるのかもしれません。しかし、FIにはそんな無駄な手順を踏む時間はありません。意思決定の遅さは、そのままプロダクトの死を意味します。私は即座にローカル環境のVS Codeを開き、TypeScriptのコードへと指を走らせました。

## 自律稼働ツール「Antigravity」のセキュリティ最適化と、一撃のデプロイ

今回、最優先で取り組んだのは、自社で開発・活用している自動化およびセキュリティ管理ツール「Antigravity」の仕様変更と最適化でした。

まず、APIキー失効とトークン切れによるエラーを検知した際、自動的に予備のAPIプロバイダや別のAPIキーへと動的にローテーションする処理が、Next.jsのEdge Runtime上でうまく機能していませんでした。TypeScriptの厳格な型定義（Strict Mode）において、ランタイム時の環境変数の型安全性が担保しきれておらず、フォールバック処理で例外を投げていたのです。

私は、LLMにエラーログと関連するTypeScriptのコードを流し込み、最も安全で型安全なフォールバック処理のパッチコードを爆速で生成させました。AIが出力したコードのセマンティクスを瞬時に脳内で検証し、手元で修正を施します。

さらに、VercelのDeploy Hookをバイパスし、デプロイが重複した際には先行するキューを自動でキャンセル（Cancel Previous Builds）する設定を、API経由で一括適用するスクリプトをその場で実装しました。

```typescript
// Vercel APIを叩き、進行中の古いデプロイを即座にキャンセルする自動化処理のコアロジック
async function cancelDuplicatedDeployments(projectId: string) {
  const deployments = await fetchVercelDeployments(projectId);
  const buildingDeployments = deployments.filter(d => d.status === 'BUILDING');
  
  if (buildingDeployments.length > 1) {
    // 最新以外のデプロイをすべてキャンセル
    for (let i = 1; i < buildingDeployments.length; i++) {
      await cancelDeployment(buildingDeployments[i].id);
      console.log(`[Canceled] Duplicated deployment target: ${buildingDeployments[i].id}`);
    }
  }
}
```

このスクリプトを「Antigravity」のセキュリティ設定に組み込み、一撃で全てのVercel環境へ反映させました。

朝8時に検知した複数プロジェクトのトリプルパンチは、11時には完全に沈静化。システムはバグを乗り越え、自動キーローテーション機能と自動重複回避機能を備えた、以前よりも遥かに強固なシステムへと生まれ変わったのです。

## 泥臭いトラブルすらも、AIで「仕組み化」する開発文化

私がこのエピソードで伝えたいのは、「私はこんなにトラブル対応が早い」という自慢話ではありません。**「泥臭いトラブルが発生した瞬間こそ、それを仕組み化し、二度と人間の手を煩わせないようにする絶好のチャンスである」**という、FIの極めて一貫した開発文化です。

私たちは「自社実証ファースト」を掲げています。
自分がヘビーユーザーとして自社システムを限界まで使い倒し、そこで出たエラーや不具合を身を以て体験する。だからこそ、顧客に提供するプロダクトの痛いところが誰よりもリアルに理解できます。

今回の障害対応でも、私はただ「動くように直した」だけではありません。
「なぜこのエラーが起きたのか」をデータドリブンに分析し、同じエラーを今後はシステム自身が自動検知して自己修復する（セルフヒーリング）仕組みを組み込みました。手動の作業は一瞬のコストであり、仕組み化は一生の資産です。

FIが求めているのは、指示書通りにきれいなNext.jsのコンポーネントを書くだけの、綺麗事のエンジニアではありません。

泥臭いバグや障害を目の前にした時、「よし、これをどうやって自動化してやろうか」と目が輝くような、野生のエンジニアリング精神を持った仲間です。スピード感に妥協せず、即断即決・即実行でデプロイまでをその日のうちにやり遂げる。そんな圧倒的な当事者意識を持つ人物と一緒に、この自動化の山を登りたいと考えています。

私たちがやっているのは、単なるWeb制作でも、流行りのAIラッパー開発でもありません。
