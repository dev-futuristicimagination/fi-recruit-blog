## 採用ブログ、なぜ突然の404？ Next.js/Vercel/GitHub APIが絡む複雑怪奇なシステムの問題

僕が運営するすべてのサイトは、基本的に同じような技術スタックで構築されています。採用ブログも例外ではなく、Next.jsで構築し、Vercelでデプロイ。そして、コンテンツはGitHubリポジトリにMarkdownファイルとして保存され、それをGitHub APIを使って取得し、ページに表示するという仕組みです。徹底的な自動化とGitHubを中心としたコンテンツ管理は、僕の会社の生命線と言っても過言ではありません。

最初に異変に気づいたのは、Vercelのビルドログでした。いつもはサッと完了するはずのビルドが、やけに時間がかかり、最終的には「Failed」の赤い文字。ログを詳しく見ていくと、GitHub APIからのデータ取得に失敗していることが分かりました。

「GitHub APIか。レートリミットかな？」

最初は単純な理由だと思いました。しかし、調査を進めるうちに、事態は想像以上に複雑であることが判明しました。GitHub APIの認証エラー、それに伴うデータ取得の失敗、VercelのCDNキャッシュが古い情報を配信し続ける問題、そしてNext.jsのビルドプロセスで、TypeScriptのStrict Mode環境下で発生していた型エラーが、連鎖的に絡み合っていたんです。

正直、頭の中はパズルのピースがバラバラになったような状態でした。複数のリポジトリとシステムを並行して管理している僕にとって、どこから手をつければいいのか、一瞬だけ迷宮入りしたような感覚でしたね。

## 泥沼のデバッグと自動化への執念：僕がシステム安定性にかける想い

しかし、立ち止まっている暇はありません。この会社の、そして僕自身の命運がかかっている。問題が発生すれば即座に原因を特定し、根本的な解決策を講じるのが僕の仕事スタイルです。

まずは、原因の特定から。一つ一つのピースを切り分け、再現条件を探しました。

1.  **GitHub API認証の再構築**: GitHub APIの仕様変更と、僕の管理している複数の認証トークンの管理方法が少し古くなっていたことが原因でした。脆弱性対応も兼ねて、より堅牢な認証フローと、定期的なトークン更新の仕組みを導入しました。
2.  **Next.jsとVercelのキャッシュ戦略の見直し**: GitHub APIからのデータ取得が不安定になった際、Vercelのキャッシュが古いコンテンツを配信し続けてしまう問題がありました。Next.jsのISR（Incremental Static Regeneration）の挙動を深く理解し直し、エラー時のフォールバックや、キャッシュの無効化戦略を最適化。これで、APIの不調が直接ユーザー体験に影響することを防ぎました。
3.  **TypeScript Strict Modeとの格闘**: 普段からTypeScriptのStrict Modeを極めて重視している僕にとって、ここで型エラーが発生していることは屈辱的でした。APIからのレスポンスの型定義が一部甘くなっていたこと、そしてエラーハンドリングが不十分だったことが原因です。既存のコードベース全体をレビューし、型安全性を徹底的に高めました。ここは僕の得意分野ですが、今回は連鎖的な影響が大きく、普段以上に慎重な対応が求められましたね。

そして、最も重要だったのが「二度と同じミスを繰り返さない」ための仕組みづくりです。僕の哲学は「自動化できるものは全て自動化」。手作業での確認なんて、もってのほかです。

*   Vercel CronとGitHub Actionsを連携させ、定期的にGitHub APIの稼働状況とデータ取得テストを自動実行。
*   異常があれば、即座に僕のDiscordに通知が飛ぶようにシステムを強化。
*   既存の記事を一括で更新するシステムにも、同様のチェック機構を追加し、潜在的なエラーを未然に防ぐ仕組みを構築しました。

この一連の対応を通じて、採用ブログは無事復旧。そして、以前よりもはるかに堅牢で安定したシステムへと生まれ変わりました。この経験は、僕の「システム品質と安定性」へのこだわりを再認識させてくれる、貴重な機会となりました。
