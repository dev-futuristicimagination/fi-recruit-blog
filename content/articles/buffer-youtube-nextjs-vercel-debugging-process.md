## 牙を剥いた多重エラー：Buffer、YouTube API、そしてVercelのタイムアウト

何が起きていたのか、技術的な詳細を明かそう。

私たちのシステムは、AIがトレンドデータを解析し、自動的にコンテンツを生成・デプロイし、さらにSNSへのシェア（Buffer経由）や関連する動画情報の取得（YouTube API経由）までを全自動で行う。

今朝の障害は、いくつかの不運な要因が重なった「ドミノ倒し」だった。

1. **Bufferのトークン期限切れ**：SNS自動投稿を担うBufferのアクセストークンが、API仕様変更の影響で予期せず失効。エラーハンドリングが不十分だったため、例外をスローしたままスレッドが停止した。
2. **YouTube APIのクォータ（上限）枯渇**：深夜のバッチ処理が異常ループを起こし、Google APIの無料枠制限に引っかかってAPIが応答を拒否。
3. **VercelのServerless Functionタイムアウト（504）**：上記2つのエラー処理がリトライを繰り返した結果、Vercelの関数実行時間制限（10秒）を超過。処理全体が途中で強制終了し、自動デプロイが中途半端な状態でロックされた。
4. **Discord通知の無限ループ**：エラーを検知した通知ボットがエラーログをDiscordに送り続け、今度はDiscordのRate Limitに引っかかり、Slack/Discord連携モジュールが完全にスタックした。

まさに「多重クラッシュ」である。どれか一つを直すだけでは、全体のパイプラインは復旧しない。データが不整合な状態で残れば、最悪の場合、サイトの一部が表示崩れを起こす危険もあった。

## 勘には頼らない。TypeScript Strict ModeとVercelログで原因の核心を突く

デバッグにおいて、私は「おそらくここだろう」という勘を一切信用しない。信頼するのは「厳格な型」と「定量的なログ」だけだ。

まずVercelのダッシュボードを開き、実行時ログ（Runtime Logs）をチェックする。タイムアウトを起こした関数の実行トレースを追い、どのコンポーネントでスタックしたかを特定した。

FIの開発環境は、Next.js 16（App Router）とTypeScript（厳格なStrict Mode）で統一されている。型の安全性を極限まで高めているため、バグの特定は極めてスムーズだ。

原因のコードは、Buffer APIのレスポンスをパースする型定義にあった。APIが返してきたエラーオブジェクトのスキーマが私の想定（Zodによるスキーマ定義）とわずかにズレていたため、パースエラー（ValidationError）が発生し、それがキャッチされずに上位の関数まで突き抜けていた。

対策として、以下の3ステップを即座に実行した。

* **例外処理の堅牢化**：Buffer APIのレスポンスパース部分にセーフティネット（`safeParse`）を噛ませ、エラー時でもデフォルトのモックデータを返して処理を続行させるように修正。
* **YouTube APIのフォールバック設計**：API上限エラーを検知した場合は、新規取得を諦め、ローカルキャッシュから過去の人気動画データを引いてくるフォールバック処理を実装。
* **Discord通知のデバウンス（流量制御）**：同じエラーが短時間に連続して発生した場合、通知を1件に集約するミドルウェアを追加。

```typescript
// エラー通知のデバウンス処理（簡易版イメージ）
const triggerNotification = debounce(async (error: Error) => {
  await discordClient.send({ content: `[Error] ${error.message}` });
}, 5000); // 5秒間は同一通知をスキップ
```

GitHubにブランチを作成し、ローカルでテストを実行。型チェックが通り、ユニットテストがすべてグリーンになったことを確認して、`main`ブランチへPush。VercelのDeploy Hookが走り、CI/CDパイプラインが自動で本番環境へとコードを反映させていく。
