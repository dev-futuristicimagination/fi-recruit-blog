しかし、自動化されているということは、一箇所が狂うとドミノ倒しのようにすべてが止まるリスクと隣り合わせだということでもある。この日、私は目覚めてすぐに、10以上のプロダクトに跨るエラー群を一人で処理しなければならないという極限状態に直面していた。

## 1人11サイト運営のリアル。ある朝届いた、容赦のないエラー通知の山

まずは状況整理だ。エラーの主な原因は3つあった。

1つ目は、SNSへの自動投稿システムで叩いていたBuffer APIの仕様変更。これにより、朝のバッチ処理で動く投稿パイプラインが完全にストップしていた。

2つ目は、SEOリライト不具合。Google Search Console APIやGA4 APIからデータを取得し、アクセス数の少ない低ランク記事を自動検出してGemini APIでリライトをかけるエンジンがあるのだが、型定義の不整合でAPI連携部分が異常終了していた。

そして3つ目が、それらの修正をデプロイしようとした瞬間に発生した、Next.jsのSSG（静的サイト生成）におけるビルドエラー。Vercel上で複数のプロジェクトがドミノ式にデプロイエラーを吐き出していた。

元SESエンジニアだった私にとって、障害対応自体は珍しいものではない。しかし、1人で11ものサイトと、それを支えるバックエンドシステムを同時に保守するのは物理的な限界がある。架空の数字や誇張した実績をアピールして自分を大きく見せるつもりはない。この時の私は、シンプルに焦っていた。午前中にはクライアントとのGemini API受託開発の打ち合わせも控えている。

この窮地を突破するために使ったのが、自作の複数エージェント並行セッション管理ツール「Antigravity」だった。

## 自社製AIアシスタント「Antigravity」の制限を解除し、秒速でバグの急所を突く

「Antigravity」は、私が自分の作業効率を極限まで引き上げるためにNext.jsとTypeScriptで構築した自作のAIアシスタントツールだ。複数のAIエージェントを同時に動かし、複雑なファイル構造の書き換えやバグ修正を並行して行わせることができる。

私は「Antigravity」のシステムセキュリティ設定を「Unrestricted（制限なし）」へと最適化した。開発中の安全弁をすべて外し、AIにプロダクトの全ソースコードへの直接的な変更・アクセス権限を与え、文字通り「脳直」でバグ修正を行わせるためだ。

ここからの作業は、まさにバイブコーディングだった。

```typescript
// エラーの原因となったTypeScriptの型定義の罠
// Strict Modeによる厳格なNullチェックをバイパスせず、頑健なエラーハンドリングへ修正
interface BufferPostResponse {
  id: string;
  success: boolean;
  mapping?: Record<string, string>; // mappingがundefinedになり得る設計を見落としていた
}
```

私はAntigravityにBuffer APIの修正ログを流し込み、仕様変更されたエンドポイントとパラメータの差分を瞬時に解析させた。AIは仕様変更箇所を特定すると、数秒で修正コードを提案してきた。

同時に、Next.jsのSSGビルドエラーについても解決を図った。エラーの原因を表面的な対処で濁すのは私の主義に反する。ログを詳細に読み解くと、SSGの静的生成時に対象記事が一時的にDBに存在しない場合の例外処理が抜けていた。いわゆるSSGの限界に起因する設計ミスだ。

「SSGの限界を考慮し、フォールバックの設定を `blocking` に変更した上で、ビルド時の動的データ取得部分に堅牢なリトライ処理を追加してくれ」

Antigravityに指示を投げる。AIは即座にコードを生成し、私はそれらをNext.jsのTypeScript Strict Modeの規約に則ってレビューした。堅牢性と品質には一切妥協しない。コードの復元と修正が完了し、Vercelへのプッシュが開始される。

次々とグリーンに変わっていくVercelのデプロイステータス。時計を見ると、エラー通知を受けてからわずか25分しか経っていなかった。10以上のプロダクトに跨る致命的なエラーは、AIアシスタントとのシームレスな連携によって、文字通りなぎ倒された。

## 「自分で使っている状態」だからこそ、プロダクトのバグは「我が事」になる

正直に言うと、私は「自分が使っていないプロダクトを開発して売る」という行為が信じられない。

「弊社自身が使っている状態を作らないと、顧客に刺さるプロダクトは作れない」

これが私の確固たる信念だ。だからこそ、Futuristic Imaginationでは、受託開発やSaaS開発の前に、まず自社でメディアを11個立ち上げ、自作のAIパイプラインを24時間稼働させ、エラーが出れば自分の手を血に染めて修正する。TypeScriptの厳格な型定義と格闘し、Next.jsの最新機能の罠を踏み抜く。そのすべての泥臭いプロセスを通過しているからこそ、私たちの技術は本物なのだ。

この朝の修羅場も、単なるトラブルシューティングではない。自社システムを自ら極限まで酷使しているからこそ得られた、貴重な実証データだ。「SSGのデプロイエラーをどう回避するか」「API仕様変更にどう柔軟に対応するか」といった現実解は、机の上の勉強会では絶対に身に付かない。

ゴールドラッシュに乗り遅れないこと。最新のAI技術やトレンドを追い、実戦で使い倒し、先行者利益を取りに行くこと。このスピード感に追いつくためには、日々の保守運用すらも「究極の効率化」の実験場でなければならない。
