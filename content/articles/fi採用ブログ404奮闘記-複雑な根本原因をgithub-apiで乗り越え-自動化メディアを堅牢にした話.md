### 最初の気づきと深まる闇 – なぜ404は起こったのか？

それは、僕が手がけるある自動運営メディアで、突如として特定の記事ページが404 Not Foundを返し始めた時のことでした。Vercel Cronで定期的にGoogle Geminiが記事を生成し、それがGitHubに自動保存され、Next.jsでビルドされてデプロイされる――この一連の自動化パイプラインは、日々たくさんのコンテンツを生み出しているはずでした。にもかかわらず、なぜか新しい記事の一部が消え、時には既存の記事までアクセスできなくなる。

最初は「またVercelがご機嫌斜めか？」と、軽く考えていました。よくあるキャッシュの問題か、デプロイに失敗したか。でも、確認してもVercelのデプロイ履歴には成功が記されている。GitHubのリポジトリを見ても、ファイルはちゃんと存在している。Discord連携で受け取る成功通知も問題ない。

「おかしい。どこかで何かが食い違っている……」

この時点で、僕の頭の中では「これは単純なバグじゃない、もっとシステムの深いところに根本原因が潜んでいるぞ」という警告灯が点滅し始めていました。なにせ、僕らは「1人で11サイト自動運営」という、普通なら考えられないような規模のシステムを回しているわけです。一つ一つのサイトは独立しているように見えて、裏側では共通のAIパイプライン、GitHubリポジトリ、Vercelデプロイ戦略が絡み合っている。この複雑な絡み合いの中に、原因の糸口があるはずだと直感しました。

自動化を極めることは、同時にデバッグを極めることでもある。この時は、まさにその真髄に触れるような気がしました。

### 複雑な根本原因の特定 – GitHub APIとの格闘

僕らの自動生成システムは、AIが記事を生成した後、それをGitHubのリポジトリに特定のパスでコミットし、その変更をトリガーにVercelがデプロイする、という流れです。今回、ファイルはGitHub上にあるのに404になる、という状況だったため、まず疑ったのは「GitHub上でのファイルの状態」と「Vercelがビルド時に参照しているGitHubの状態」の間に乖離があるのではないか、ということでした。

実は、ここで一番苦労したのは、情報が点在しすぎていたことでした。GitHubのWeb UIでは限界があります。コミット履歴を一つ一つ追うだけでは、ファイルの細かいパス変更や、本当にそのタイミングでファイルが正確にコミットされたのか、さらにはそのファイルがビルド時に正しく認識されたのか、といった深層は掴めません。

そこで僕が取り組んだのが、**GitHub APIの徹底的な活用**でした。
VercelのデプロイログとGitHubのコミット履歴を突き合わせるだけではなく、直接GitHub APIを叩いて、以下の点を徹底的に洗い出しました。

1.  **特定コミット時点でのリポジトリ構造の再現:**
    `GET /repos/{owner}/{repo}/git/trees/{sha}?recursive=1` を使い、特定のコミットハッシュ（sha）時点でのリポジトリ全体のファイルパスとBlobハッシュを取得。これにより、Vercelがビルドを開始したコミットで、本当に目的のファイルが存在していたのか、そのパスは正しかったのかを正確に検証しました。
2.  **ファイルのBlob内容の比較:**
    疑わしいファイルのBlobハッシュを使って `GET /repos/{owner}/{repo}/git/blobs/{file_sha}` で直接ファイル内容を取得し、意図しない変更や破損がないか確認。
3.  **Webhookイベントログの解析:**
    VercelがGitHubからのWebhookを受けてデプロイを開始するタイミングと、GitHub側のコミットが完全に同期しているかを `GET /repos/{owner}/{repo}/hooks` で取得できるログなどから確認。

この調査で浮かび上がったのは、複数の要因が絡み合った、ある種の「タイミングの問題」でした。

AIが生成した記事のファイル名に、ごく稀にNext.jsのルーティングと相性の悪い特殊文字が含まれてしまうことがあった。
→ GitHubにはコミットされるが、Vercelのビルドプロセス（特にNext.jsのファイルシステムルーティングの解釈）でエラーを起こしていた。
→ 加えて、非常に短い間隔で複数の記事が生成・コミットされた際に、VercelのWebhook処理とGitHubの内部的な更新がごくわずかにずれ、古いリポジトリの状態が参照されてビルドされてしまうケースがあった。

これらの複雑な要因が組み合わさることで、「GitHubにはあるのに、Vercelで404になる」という不可解な現象を引き起こしていたのです。TypeScriptのStrict Modeで鍛え上げたデバッグスキルも、GitHubとVercelの連携という複雑な環境下では、一筋縄ではいきませんでした。

### 堅牢な自動化メディアへの進化 – 得られた教訓と未来

この一件で、僕の自動化に対する考え方はさらに深まりました。単に「動けばいい」ではない、「いかに堅牢に、予測不能な事態にも耐えうるか」という視点です。

得られた根本原因に基づき、いくつかの抜本的な改善策を講じました。

1.  **AI生成記事のファイル名バリデーション強化:**
    AIが生成した記事のファイル名やパスに、Next.jsのルーティングで問題を引き起こす可能性のある特殊文字が含まれていないかを、GitHubへのコミット前に厳しくチェックする仕組みを導入しました。もし問題があれば、自動的に修正するか、警告を出すようにしました。
2.  **GitHub APIによるデプロイ後の整合性チェック:**
    Vercel Cronの実行後、Vercelのデプロイが成功したことをトリガーに、再度GitHub APIを使ってデプロイされたサイトのファイルパスリストと、本来GitHubにあるべきファイルパスリストを比較する仕組みを導入。もし不整合があれば、Discordにエラー通知を飛ばし、場合によっては自動でリデプロイを試みるようにしました。
3.  **GitHub Actionsを活用した定期的なシステム健全性チェック:**
    週に一度、GitHub Actionsでスクリプトを実行し、リポジトリ全体のファイル整合性や、各サイトの主要ページが404を返していないかをチェックする仕組みを構築。問題があれば即座にDiscordに通知します。

これらの改善により、僕が運営するAIメディアは以前にも増して堅牢なシステムへと進化しました。今では、安心して「バイブコーディング」で新しい機能開発に集中できる環境が整っています。
